# 测试、文档与输出规则

# 必跑验证

常规代码或规则变更至少执行：

```txt
cargo fmt --check
cargo test
cargo run -- validate
cargo run -- preflight
cargo run -- errors --current --open
```

纯文档变更也必须执行 `cargo run -- validate` 和中文文档检查相关验证。

# 测试覆盖

风险越高，测试越完整。共享行为、跨模块契约和用户可见流程必须覆盖：

1. 单元测试。
2. 边界测试。
3. 错误路径测试。
4. 必要时覆盖跨平台行为。

# 文档规则

1. Markdown 必须中文可读。
2. 禁止英文占位文档。
3. 禁止乱码文档。
4. 禁止 Emoji。
5. `Agents.md` 只保留入口规则和按需阅读索引。
6. 详细设计和长规则写入 `forge/architecture/`。

# 输出规则

每轮最终说明应简洁包含：

1. 当前目标。
2. 修改内容。
3. 验证结果。
4. 当前分支、本地提交信息，以及是否已进入统一 push 和 PR 阶段。
5. 未完成事项或阻断原因。
6. 任务进度。
7. 未完成任务列表。

不要把大段命令输出原样贴给用户，只汇总关键结论。

# 进度格式

最终回复结尾必须追加：

```md
任务进度：
- 已完成：...
- 进行中：...
- 未完成：...

未完成任务列表：
- ...
```

若没有未完成任务，写 `无`。

# 阶段性回复规则

1. 任务组未被用户确认完成时，最终回复必须明确写出“未 push、未创建 PR”。
2. 若本轮只完成部分任务，必须列出剩余任务，并说明后续对话继续使用当前分支。
3. 若任务组已确认完成并创建 PR，必须列出 PR 地址、关联 Issue、版本提升和远端分支删除状态。
4. 禁止把“本轮提交完成”描述成“任务组已完成”，除非用户已经确认不再继续微调。
5. 未提升版本的本地提交必须说明“未提升版本，因此提交信息不携带版本号”。
