# 版本信息
- 版本号：v2
- 时间：2026-04-29
- 父版本：v1

# 目标

优先推进 SelfForge 的自我进化能力，完成一个最小但可验证的候选版本生成闭环：从当前稳定版本 v1 生成 v2 候选版本，写入独立 workspace、forge 归档文档和持久化状态，但不执行最终版本切换。

# 计划（Plan）

1. 读取 v1 记忆、任务、版本记录和 state/state.json。
2. 明确本轮边界：只生成候选版本，不实现完整进程启动、切换和回滚。
3. 新增结构化状态读写能力。
4. 新增 EvolutionEngine，负责验证当前版本、计算下一版本、生成候选版本归档并更新状态。
5. 将 Supervisor 接入自我进化入口，并在 CLI 增加 evolve 命令。
6. 编写单元测试、边界测试和错误测试。
7. 执行格式化、编译检查、测试、运行时验证和真实 evolve。
8. 记录错误、版本变化和可复用经验。

# 执行过程

1. 读取当前仓库结构，确认 v1 已具备 init、validate、status 三个 CLI 命令。
2. 读取 forge/memory/v1.md、forge/tasks/v1.md、forge/versions/v1.md 和 state/state.json。
3. 写入 forge/tasks/v2.md，确定 v2 的任务目标与计划。
4. 新增 src/state.rs，使用 serde_json 结构化读写 state/state.json。
5. 新增 src/evolution.rs，实现候选版本生成、版本号递增、候选验证和状态更新。
6. 更新 src/layout.rs，使 SelfForge 支持指定版本号，并为后续候选版本生成动态模板。
7. 更新 src/runtime.rs、src/supervisor.rs、src/main.rs，接入 verify_version、prepare_next_version 和 evolve CLI。
8. 执行 cargo fmt、cargo check、cargo test、cargo run -- validate。
9. 执行 cargo run -- evolve "优先推进自我进化候选版本生成"，生成 workspaces/v2 与 forge v2 文档，并把 state 推进到 candidate_prepared。

# 评估

v2 已具备受控自我进化的第一段能力：能够在不破坏当前稳定版本的前提下生成下一候选版本，并将候选状态持久化。当前仍未实现真正的沙箱执行、资源限制、候选进程启动、并行验证、切换和回滚，因此 state 保持 current_version=v1，status=candidate_prepared。

# 优化建议

1. 下一版优先实现候选版本进程启动接口，但仍保持 current_version 不变。
2. 再下一步实现 v1 与 v2 的并行验证协议。
3. 在验证协议稳定后，再实现切换与回滚状态迁移。
4. 清理旧 v1 文档模板的编码历史问题，避免长期保留兼容性噪音。

# 可复用经验

1. 自我进化应先生成候选版本和审计记录，再考虑运行进程替换。
2. state/state.json 必须保留稳定版本与候选版本的区别，避免未验证候选误升为当前版本。
3. 对版本号递增要单独测试边界和错误输入，防止未来状态迁移误判。
4. CLI 的 evolve 命令应该只做一个可验证阶段，不把启动、验证、切换、回滚混在一次大改里。
