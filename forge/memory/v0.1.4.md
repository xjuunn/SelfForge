# 版本信息
- 版本号：v0.1.4
- 时间：2026-04-29
- 父版本：v0.1.3

# 目标

实现 SelfForge 的最小可运行闭环：候选版本存在时，可以通过一个命令完成当前版本验证、候选版本验证、成功提升或失败回滚，并将结果持久化到状态文件。

# 计划（Plan）

1. 读取 `v0.1.1`、`v0.1.2`、`v0.1.3` 的历史记忆。
2. 设计最小闭环，只覆盖候选验证、提升和回滚，不扩展到沙箱资源限制。
3. 在进化引擎中新增显式回滚能力。
4. 在进化引擎中新增候选运行闭环。
5. 在 Supervisor 和 CLI 中暴露 `cycle` 与 `rollback` 命令。
6. 增加测试覆盖成功提升、手动回滚和候选验证失败回滚。
7. 运行格式化、编译、测试和 Runtime 验证。
8. 使用真实 `cycle` 提升 `v0.1.3`，再生成 `v0.1.4` 候选。

# 执行过程

1. 在 `src/evolution.rs` 中新增 `RollbackReport`、`CycleResult`、`CycleReport`。
2. 实现 `rollback_candidate`：清空候选字段，保留当前稳定版本，将状态写为 `rolled_back`。
3. 实现 `run_candidate_cycle`：先验证当前版本，再验证候选版本；成功则调用提升，失败则调用回滚。
4. 在 `src/supervisor.rs` 暴露闭环运行与回滚入口。
5. 在 `src/main.rs` 新增 `cycle` 与 `rollback [reason]` 命令。
6. 在测试中构造破坏候选工作区的场景，确认失败时不会切换当前版本。
7. 执行 `cargo run -- cycle`，真实提升 `v0.1.3`。
8. 执行 `cargo run -- evolve --patch "实现最小候选运行闭环"`，生成 `v0.1.4` 候选。

# 评估

`v0.1.4` 让 SelfForge 具备最小运行闭环：系统不再只依赖人工分别执行验证和提升，而是可以通过 `cycle` 自动完成候选验证后的受控切换或回滚。当前仍不是完整自进化系统，尚未实现沙箱资源限制和 AI 代码生成，但已经具备可运行、可验证、可回滚的最小骨架。

# 优化建议

1. 下一步实现候选验证报告持久化，记录每次 `cycle` 检查了哪些路径。
2. 之后实现工作区级沙箱执行和超时控制。
3. 再之后实现失败时自动写入结构化错误文件。

# 可复用经验

1. 最小可运行版本应先闭合状态机，再扩展复杂执行能力。
2. 回滚不应删除候选工作区，候选文件本身是审计资料。
3. 自动闭环必须先验证当前稳定版本，避免在基线已损坏时继续推进候选。
