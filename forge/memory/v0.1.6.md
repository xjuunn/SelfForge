# 版本信息
- 版本号：v0.1.6
- 时间：2026-04-29
- 父版本：v0.1.5

# 目标

建立更合理、更专业、更方便扩展的 SelfForge 项目结构，让 AI 后续开发时有明确目录边界、应用入口和架构规则；同时继续推进最小可运行闭环，让用户可以通过一个简单命令完成候选推进。

# 计划（Plan）

1. 读取最近版本记忆和当前状态。
2. 提升已验证的 `v0.1.5` 候选版本为当前稳定版本。
3. 写入 `v0.1.6` 任务文档和项目结构架构文档。
4. 新增 `src/app/` 应用层目录。
5. 实现 `SelfForgeApp::advance` 作为最小闭环应用服务。
6. 在 CLI 中新增 `advance [goal]`，降低使用难度。
7. 更新 `Agents.md`，明确后续功能优先进入应用层。
8. 增加测试覆盖无候选生成、成功提升后生成下一候选、失败回滚后停止。
9. 执行格式化、编译、测试、Runtime 验证和真实 `advance` 验证。

# 执行过程

1. 执行 `cargo run -- cycle`，将 `v0.1.5` 提升为当前稳定版本。
2. 新增 `forge/architecture/project-structure.md`，记录根目录、源码分层、扩展规则和 Git 规则。
3. 新增 `.gitattributes`，约束 Rust、Markdown、JSON、TOML 的文本换行规则。
4. 扩展 `.gitignore`，忽略常见本地系统文件和 Rust 备份文件。
5. 新增 `src/app/mod.rs` 与 `src/app/minimal_loop.rs`。
6. 新增 `SelfForgeApp`、`MinimalLoopOutcome`、`MinimalLoopReport`。
7. 新增 `advance` CLI 命令。
8. 将当前版本常量推进到 `v0.1.5`，包版本推进到 `0.1.6`。
9. 新增 3 个应用层闭环测试，测试总数提升到 21 个。
10. 执行 `cargo run -- advance "建立应用层架构与最小闭环入口"`，生成 `v0.1.6` 候选版本。

# 代码变更

- `src/app/`：新增应用用例层，承载最小闭环业务入口。
- `src/main.rs`：新增 `advance` 命令，并将 CLI 保持为参数解析与输出适配层。
- `src/lib.rs`：导出应用层类型，更新当前版本常量和测试。
- `Cargo.toml`：包版本同步到 `0.1.6`。
- `.gitattributes`：新增仓库文本规范。
- `.gitignore`：补充本地系统文件忽略规则。
- `Agents.md`：写入源码分层和应用层扩展规则。

# 测试结果

- `cargo fmt --check`：首次失败，执行 `cargo fmt` 后通过。
- `cargo check`：通过。
- `cargo test`：通过，21 个测试全部通过。
- `cargo run -- validate`：通过，SelfForge `v0.1.5` 校验 19 个必需路径。
- `cargo run -- advance "建立应用层架构与最小闭环入口"`：通过，生成 `v0.1.6` 候选版本。

# 错误总结

- `forge/errors/v0.1.6/error-001.md`：格式化检查失败，已执行 `cargo fmt` 修复。

# 评估

`v0.1.6` 没有进行危险的大规模搬迁，而是先建立新的应用层入口和项目结构规则，属于安全的架构演进。现在 SelfForge 的扩展路径更清楚：新功能优先进入 `src/app/`，CLI 不再堆叠业务流程，底层模块继续保持单一职责。

# 优化建议

1. 下一步可以把 `lib.rs` 中的大型测试模块迁移到 `tests/` 集成测试目录，进一步改善源码结构。
2. 后续可将 `runtime.rs`、`evolution.rs` 等大文件逐步拆成目录模块，但每次只迁移一个边界。
3. 将执行报告持久化作为下一个功能步，继续提高审计能力。

# 可复用经验

1. 架构重整不应直接大搬家，应先建立新入口，再逐步迁移调用。
2. 应用层用例可以显著降低 AI 后续误用底层模块的概率。
3. 一个简单的 `advance` 入口比让用户手动组合 `cycle` 和 `evolve` 更不容易出错。
