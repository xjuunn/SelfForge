# 版本信息
- 版本号：v0.1.3
- 时间：2026-04-29
- 父版本：v0.1.2

# 目标

将小版本更新的版本记录收敛到同一个 minor 系列文件中，避免每次 patch 都生成独立版本文件；同时继续保持语义化版本号、中文文档、可验证候选和审计链。

# 计划（Plan）

1. 读取最近版本记忆、状态文件和现有版本记录。
2. 将版本记录路径从 `vMAJOR.MINOR.PATCH.md` 调整为 `vMAJOR.MINOR.md`。
3. 让候选版本生成流程追加写入系列版本文件，避免创建 patch 级版本文件。
4. 补充单元测试、边界测试和错误测试。
5. 更新 `Agents.md`，写入小版本系列单文件规则。
6. 将历史 patch 版本记录合并到 `forge/versions/v0.1.md`。
7. 将旧式 `v1/v2` 版本记录合并到 `forge/versions/legacy.md`。
8. 执行格式化、编译、测试和 Runtime 验证。
9. 提升 `v0.1.2` 为当前版本，并生成 `v0.1.3` 候选版本。

# 执行过程

1. 确认当前状态为 `current_version=v0.1.1`、`candidate_version=v0.1.2`。
2. 新增版本系列文件名计算能力，`v0.1.9` 会解析为 `v0.1.md`。
3. 更新布局校验，使当前版本只要求 `forge/versions/v0.1.md`。
4. 更新候选文档生成，使 patch 更新追加到 `forge/versions/v0.1.md`。
5. 新增 `forge/versions/v0.1.md` 与 `forge/versions/legacy.md`，保留旧版本有效内容。
6. 更新 `Agents.md` 的归档规则，明确 patch 更新禁止创建独立版本记录文件。
7. 执行测试时修复格式化差异和借用错误。
8. 执行 `cargo run -- promote`，将 `v0.1.2` 提升为当前版本。
9. 执行 `cargo run -- evolve --patch "合并小版本记录并推进最小可运行版本"`，生成 `v0.1.3` 候选版本。

# 代码变更

- `src/version.rs`：新增 major、minor、patch 访问方法、系列名和系列文件名计算函数。
- `src/layout.rs`：版本记录必需文件从完整 patch 文件改为 minor 系列文件。
- `src/evolution.rs`：候选版本记录改为追加写入系列文件，并避免重复追加同一版本条目。
- `src/lib.rs`：更新当前稳定版本常量和测试断言，新增系列版本文件测试。
- `Cargo.toml`：包版本同步到 `0.1.3`。
- `Agents.md`：写入小版本系列单文件规则。

# 评估

`v0.1.3` 完成了版本记录收敛，减少了 patch 更新产生的文件数量，并保持了审计可追溯。当前系统仍处于受控候选状态：`v0.1.2` 是当前稳定版本，`v0.1.3` 是已验证候选版本。

# 优化建议

1. 下一步实现最小可运行闭环命令，将候选验证、报告持久化、成功提升和失败回滚串联起来。
2. 继续保持 patch 级小步演进，避免过早升级 minor 或 major。
3. 旧版本文件应在明确确认后删除，删除前必须确保内容已合并。

# 可复用经验

1. patch 更新属于同一 minor 系列，应追加到同一个版本记录文件。
2. 自动种子文件可能先于候选文档创建，生成流程应避免覆盖人工文档，但记忆文件需要后续补齐。
3. 版本路径是审计结构的一部分，必须同时更新代码、规则文档和测试。
