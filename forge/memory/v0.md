# 目标

解决长期 Markdown 记忆无限增长的问题，新增热记忆压缩、冷归档和进化后的自动压缩能力。

# 计划（Plan）

1. 读取最近记忆和当前状态。
2. 编写任务文档。
3. 在记忆模块实现压缩归档用例。
4. 在 CLI 增加 `memory-compact`。
5. 让 `agent-evolve` 在提升成功后自动压缩热记忆。
6. 补充单元、边界和错误测试。
7. 执行真实命令验证、候选提升、文档归档和提交。

# 执行过程

先实现可重复执行的压缩函数，将 `forge/memory/v0.md` 中超出保留窗口的完整小节迁移到 `forge/memory/archive/v0.md`。随后将命令接入 CLI，并把 `agent-evolve` 的成功提升流程扩展为自动执行热记忆压缩。

# 代码变更

1. `src/app/memory.rs` 新增记忆压缩报告、错误类型、热记忆渲染和冷归档渲染。
2. `src/app/minimal_loop.rs` 新增应用层压缩入口，并在 `agent-evolve` 成功提升后自动压缩热记忆。
3. `src/main.rs` 新增 `memory-compact` 命令和输出。
4. `src/lib.rs` 新增压缩迁移、幂等执行、错误参数和自动压缩相关测试。
5. `Agents.md` 与 `README.md` 补充记忆压缩规则和命令。

# 测试结果

1. `cargo fmt --check` 通过。
2. `cargo test` 通过，111 个测试全部成功。
3. `cargo run -- memory-compact --current --keep 8` 通过，首次将 25 条旧记忆迁入冷归档。
4. `cargo run -- memory-insights --current --limit 5` 通过，只读取热记忆文件中的近期来源。
5. `cargo run -- agent-evolve "新增记忆压缩冷归档能力"` 通过，当前稳定版本提升为 `v0.1.40`。

# 错误总结

本轮没有未解决错误。并行执行两个 `cargo run` 时出现 Cargo 文件锁等待提示，命令最终成功，不构成系统错误。

# 评估

v0.1.40 将记忆系统从“单文件持续追加”推进到“热记忆加冷归档”的结构，既保留审计追溯能力，又降低默认 AI 上下文读取成本。自动压缩接入 `agent-evolve` 后，后续进化不会继续让热记忆无限增长。

# 优化建议

下一步应继续完善 Agent 多步自动执行，让系统在受控工具范围内自动推进多个计划步骤，并在遇到需要外部输入或验证失败时暂停、归档并给出清晰原因。

# 可复用经验

记忆读取和记忆压缩要分离：默认读取只访问热记忆，冷归档只服务审计和复盘。进化流程中的维护动作应放在应用用例层，由 `agent-evolve` 统一编排，避免 CLI 或 Agent 业务逻辑各自实现一套规则。
