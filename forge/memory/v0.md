1. 闸门记录应只做判定和归档，不应同时执行下一阶段动作。
2. 任务草案进入补丁生成前必须检查目标版本、验收检查和后续命令，避免把不完整草案传给 AI。
3. 阻断记录也有价值：它能明确告诉后续 Agent 应该修复草案，而不是继续生成补丁。

# 优化建议

下一步应实现基于任务草案的审计闸门，把任务草案验证为可执行计划后，再允许进入 `agent-patch-draft` 或后续源码覆盖流程。

# 可复用经验

1. 总结、草案、真实任务必须分层：总结保存事实和建议，草案保存待审计意图，真实任务只能由后续闸门批准后生成。
2. 状态文件必须使用无 BOM UTF-8 写入；涉及 JSON 状态的批量更新后，应立即运行 `preflight` 验证解析和布局。
3. 自动化越靠近真实修改，越需要独立记录和只读输入，避免一步跨过审计边界。
