## v0.1.7

### error-001

# 错误信息

`cargo check` 首次执行时报出 `E0308` 类型不匹配：`EvolutionError::Io` 需要 `PathBuf`，但 `append_record` 中传入了 `&Path`。

# 出现阶段

代码编译验证。

# 原因分析

`append_record` 接收 `&Path`，读取文件失败时错误记录需要拥有路径所有权，旧写法使用 `path.clone()` 只能得到 `&Path`。

# 解决方案

将错误路径改为 `path.to_path_buf()`，并保持 `fs::read_to_string(path)` 使用借用路径读取。

# 是否已解决

是。后续 `cargo check` 和 `cargo test` 均已通过。

## v0.1.10

# 错误信息

首次测试时，`runtime_executes_command_inside_version_workspace` 使用未规范化的 temp 路径断言 `run_dir`，而 Runtime 返回的是 canonical workspace 路径，导致断言失败。

# 出现阶段

单元测试验证。

# 原因分析

Runtime 为了防止工作区逃逸，会对 workspace 执行 canonicalize。测试直接使用 temp 根目录拼接路径，没有与 Runtime 返回路径保持同一规范。

# 解决方案

将测试改为验证 `run_dir` 位于 `report.workspace` 下，并检查父目录为 `runs`，不再混用未规范化路径。

# 是否已解决

是。修复后 `cargo test` 通过，24 个测试全部成功。

## v0.1.22

# 错误信息

用户反馈在 PowerShell 中设置 `SELFFORGE_AI_PROVIDER=deepseek` 后运行 `ai-request` 仍提示没有可用 AI 提供商。

# 出现阶段

AI 请求构建前置配置检查。

# 原因分析

# 解决方案

改进 `MissingProvider` 和 `MissingApiKey` 错误消息，直接提示 PowerShell 当前会话设置方式、对应 API Key 变量和 `cargo run -- ai-config` 检查命令，并补充测试覆盖。

# 是否已解决

是。缺少 DeepSeek Key 的真实命令现在会明确提示设置 `$env:DEEPSEEK_API_KEY="你的密钥"`，设置临时 Key 后 `ai-request` 可生成脱敏请求摘要。

## v0.1.23 用户反馈错误

# 错误信息

用户反馈：项目根目录 `.env` 中配置 AI 提供商和密钥后，`cargo run -- ai-request "你好"` 仍提示没有可用 AI 提供商。

# 出现阶段

AI 请求构建前的配置读取阶段。

# 原因分析

上一版本只读取当前进程环境变量，没有自动读取项目根目录 `.env`。因此当用户把 `SELFFORGE_AI_PROVIDER` 和 `DEEPSEEK_API_KEY` 写入 `.env`，但未在 PowerShell 当前会话中显式设置环境变量时，AI Provider 注册表无法看到配置。

# 解决方案

在 AI Provider 模块中新增项目环境查找器，合并进程环境变量和项目根目录 `.env`。真实进程环境变量优先，`.env` 作为本地补充配置。`SelfForgeApp::ai_config` 和 `SelfForgeApp::ai_request` 已统一改用该入口，并补充测试。

# 是否已解决

是。`cargo run -- ai-config` 已能从 `.env` 识别 DeepSeek，`cargo run -- ai-request "你好"` 已能生成脱敏请求摘要，`cargo test` 57 个测试全部通过。
