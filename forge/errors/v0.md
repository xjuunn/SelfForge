## v0.1.7

### error-001

# 错误信息

`cargo check` 首次执行时报出 `E0308` 类型不匹配：`EvolutionError::Io` 需要 `PathBuf`，但 `append_record` 中传入了 `&Path`。

# 出现阶段

代码编译验证。

# 原因分析

`append_record` 接收 `&Path`，读取文件失败时错误记录需要拥有路径所有权，旧写法使用 `path.clone()` 只能得到 `&Path`。

# 解决方案

将错误路径改为 `path.to_path_buf()`，并保持 `fs::read_to_string(path)` 使用借用路径读取。

# 是否已解决

是。后续 `cargo check` 和 `cargo test` 均已通过。

## v0.1.44

# 错误信息

开发阶段出现三个已解决错误：新增 Agent 步进任务同步后，首次 `cargo test` 编译失败，提示 `AgentStepStatus::Running` 分支未覆盖；修复后第二次 `cargo test` 有 1 个测试失败，原因是协作任务完成摘要没有包含工具标识；真实验证恢复任务板时发现完成后释放任务会残留旧完成时间。

# 出现阶段

实现与测试验证。

# 原因分析

1. 步骤状态枚举包含 `Pending`、`Running`、`Completed`、`Failed`，新增任务完成逻辑只处理了 `Pending`、`Completed`、`Failed`。
2. 任务板完成摘要只写入自然语言结果，测试和审计需要明确记录触发任务完成的工具标识。
3. `release` 通常用于已领取未完成任务，完成后释放的恢复场景暴露出 `completed_at_unix_seconds` 没有被清空。

# 解决方案

1. 将 `AgentStepStatus::Running` 纳入安全分支，保持任务状态不变。
2. 将任务完成和释放摘要改为包含步骤号、工具标识和执行摘要。
3. 在释放任务时清空 `completed_at_unix_seconds`，并补充完成后释放的测试。
4. 重新执行 `cargo fmt` 和 `cargo test`，确认 124 个测试全部通过。

# 是否已解决

是。

## v0.1.61

# 错误信息

本轮开发出现一个已解决错误：使用 PowerShell 批量更新 `state/state.json` 后，文件被写入 UTF-8 BOM，导致 `cargo run -- agent-patch-source-task-drafts --current --limit 5` 读取状态文件时提示 `expected value at line 1 column 1`。

# 出现阶段

开发、测试、验证和预检阶段。

# 原因分析

PowerShell `Set-Content -Encoding UTF8` 在当前环境中写入了带 BOM 的 UTF-8。JSON 解析器不接受首字节 BOM，因此应用命令在解析 `state/state.json` 时失败。该问题属于本轮版本文件更新过程引入的编码问题，不是任务草案业务逻辑缺陷。

# 解决方案

已使用无 BOM UTF-8 重新写入 `state/state.json`、`Cargo.toml`、`Cargo.lock`、`README.md` 和 `src/lib.rs`，并重新执行 `cargo run -- preflight`、`cargo run -- agent-patch-source-task-drafts --current --limit 5` 和 `cargo run -- errors --current --open --limit 5`，均通过。后续批量写入项目文件应优先使用无 BOM UTF-8，避免状态文件解析失败。

# 是否已解决

是。
